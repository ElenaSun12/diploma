# План тестирования и автоматизации Android-приложения «В хосписе»

## 1. ЦЕЛЬ
- Проверить Android-приложение на соответствие требованиям функциональности посредством ручного и автоматизированного тестирования.
- Проверка UI/UX соответствия требованиям
- Автоматизация критичных сценариев

## 2. ОПИСАНИЕ ПРИЛОЖЕНИЯ
Приложение «В хосписе» предназначено для работы с новостями хосписа и включает в себя: 
- информацию о новостях и функционал для работы с ними;
- тематические цитаты;
- информацию о приложении.

Основные модули:
- Авторизация и регистрация;
- Новости (список, создание, редактирование);
- Цитаты;
- Информация о приложении.

### Что надо тестировать?
- Авторизация (поля ввода, валидация)
- Функционал новостей (сортировка, добавление новости)
- Цитаты (отображение, навигация)
- UI/UX (навигация, отзывчивость интерфейса)
- Производительность (время отклика)

###  Что будут тестировать?
- Функциональные тесты (ручные + автоматизированные)
- UI-тесты (автоматизированные)
- Smoke-тесты (ручные)

###  Как будут тестировать?
- **Ручное тестирование**: исследовательское, проверка новых функций
- **Автоматизированное тестирование**: Espresso для UI-тестов
- **Инструменты**: Android Studio, Espresso, Allure, Git

###  Когда будут тестировать?
- После каждого коммита разработчика (регрессионные тесты)
- Перед релизом (полное тестирование)
- При добавлении нового функционала

### Какие критерии начала тестирования?
1. Приложение успешно собирается в Android Studio
2. Все зависимости загружены
3. Эмулятор/устройство готово к работе
4. Тестовые данные подготовлены (login2/password2)

### Какие критерии окончания тестирования?
1. Все тест-кейсы с высоким приоритетом выполнены
2. Критические баги исправлены
3. Автоматизированные тесты проходят успешно
4. Достигнут запланированный уровень покрытия (70%)

## 4. ТЕСТОВЫЙ СТЕНД:
Android Studio → AVD (Pixel 5, API 29) → Приложение

### Необходимое оборудование и ПО
- **Оборудование**: ПК, Android-устройство/эмулятор
- **Программное обеспечение**:
    - Android Studio с Android SDK
    - Espresso для UI-тестов
    - Allure для отчетов
    - Git для контроля версий

## 5. ТЕХНИКИ ТЕСТ-ДИЗАЙНА С ОБОСНОВАНИЕМ

### 5.1 Эквивалентное разделение
- **Входные данные**: валидные/невалидные логин-пароль
- **Обоснование**: проверка валидации полей ввода

### 5.2 Анализ граничных значений
- **Поля ввода**: пустые строки, максимальная длина
- **Обоснование**: предотвращение crashes при некорректных данных

### 5.3 Таблица принятия решений
- **Сценарии авторизации**: 2^2 = 4 комбинации
- **Обоснование**: полное покрытие логики авторизации

## 6. РИСКИ И ПУТИ ИХ РАЗРЕШЕНИЯ

### Риски при автоматизации:
1. **Изменение UI элементов** → Использовать стабильные ID, Page Object Pattern
2. **Падение тестов на разных устройствах** → Тестировать на Pixel 5 (референс)
3. **Долгое выполнение тестов** → Параллельный запуск, оптимизация
4. **Сложность поддержки тестов** → Четкая структура, комментарии

### Пути разрешения:
- Регулярный рефакторинг тестов
- Использование CI/CD для раннего обнаружения проблем
- Документация всех тестовых сценариев

## 7. СПЕЦИАЛИСТЫ ДЛЯ АВТОМАТИЗАЦИИ
1. **QA инженер** - написание и поддержка тестов
2. **Разработчик** - помощь с ID элементов, архитектурой
3. **DevOps инженер** - настройка CI/CD (опционально)

## 8. ИНТЕРВАЛЬНАЯ ОЦЕНКА С УЧЁТОМ РИСКОВ

### Фазы работы:
- **Неделя 1**: Исследование, ручное тестирование, документация
- **Неделя 2**: Настройка проекта, написание первых автотестов
- **Неделя 3**: Автоматизация основных сценариев, Allure отчеты
- **Неделя 4**: Финальная проверка, оптимизация, документация

### С учетом рисков (+20% времени):
- Итого: **4 недели** × 1.2 = **~5 недель**
